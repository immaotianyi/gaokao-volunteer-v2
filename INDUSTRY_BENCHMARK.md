# 传统高考志愿填报 AI 产品 — 系统架构总览

> 本文对市面上主流高考志愿填报 AI 产品的通用技术架构进行归纳分析，
> 用于与「高考志愿狙击手」做技术对比参照。

---

## 一、通用架构总览

市面主流志愿填报 AI 产品遵循一套高度同质化的架构范式：

```
用户 (微信小程序 / App / 网页)
    │
    ▼
前端界面 (填写分数、省份、选科)
    │
    ▼
后端 API 网关 (单一请求-响应模式)
    │
    ├──▶ 数据库查询层（结构化查询）
    │      │  历年录取分数线表
    │      │  位次对照表
    │      │  招生计划表
    │      │  SELECT ... WHERE score BETWEEN ... AND ...
    │      │
    │      └──▶ 输出：可报院校列表 + 录取概率
    │
    └──▶ 规则引擎（硬编码）
           │  体检规则：色盲 → 排除医学类
           │  选科规则：历史类 → 排除物理类
           │  批次规则：本科线 → 过滤专科
           │
           └──▶ 输出：过滤后的志愿方案
```

## 二、产品分层解剖

### 2.1 数据层 — 静态数据库查询

**核心逻辑**：

```
输入：考生分数 X，省份 P，科类 S
处理：
    1. 查表获取位次：R = lookup_rank(P, S, X)
    2. 查表获取可报院校：SELECT university, major FROM admission_history
                          WHERE province = P
                            AND subject = S
                            AND lowest_score <= X + tolerance
                            AND lowest_score >= X - tolerance
                          ORDER BY lowest_score DESC
    3. 计算"录取概率"：probability = f(X, historical_score, trend)
    4. 排序输出
```

**数据来源**：
- 各省考试院历年投档线（公开数据）
- 教育部学科评估结果（用于院校推荐排序）
- 各高校招生计划（公开数据）

**数据更新频率**：年度更新，通常在高考前集中灌库一次。

**本质**：一个带 UI 的 **SQL 数据库查询应用**。

---

### 2.2 推荐层 — 简单规则匹配

**典型实现**：

```
# 伪代码
def recommend(score, province, subject):
    universities = []
    for u in all_universities:
        if u.min_score <= score <= u.max_score:
            if u.subject == subject:
                prob = (score - u.min_score) / (u.max_score - u.min_score)
                universities.append({u, prob})
    return sorted(universities, by prob, descending=True)
```

**录取概率公式**（市面通用）：

```
概率 = (考生分数 - 该校最低录取分) / (该校最高录取分 - 最低录取分)
```

- 或基于位次差值：`概率 = f(考生位次, 该校最低位次)`
- 本质是简单的**线性插值**，无任何 AI 参与

**冲/稳/保分级**（市面通用算法）：

```
冲：高于最低录取分 0~10分    → 概率 30%~60%
稳：高于最低录取分 10~30分   → 概率 60%~90%
保：高于最低录取分 30分以上   → 概率 >90%
```

---

### 2.3 规则引擎 — 硬编码排除

**典型实现**：

```
规则 1: IF 选科 = 历史类 THEN 排除所有要求物理的专业
规则 2: IF 体检 = 色盲 THEN 排除"临床医学"+"口腔医学"+"麻醉学"
规则 3: IF 体检 = 色弱 THEN 排除"药学"+"化学类"
规则 4: IF 分数 < 本科线 THEN 仅显示专科批次
规则 5: IF 语种 != 英语 THEN 排除标注"仅限英语"的院校
```

**覆盖范围**：通常只覆盖 5~10 条通用规则，由开发团队人工编写，不读取招生章程原文。

**盲区**：
- 各高校招生章程中的**个性化补充条款**全部忽略
- 对"色弱不宜报考""单色识别不全者不予录取"等细分条款不区分
- 不包含单科成绩门槛检测
- 不包含专业组内调剂风险评估

---

### 2.4 商业变现层 — 付费墙

市面产品的主流变现模型：

```
免费功能：
  - 查看可报院校列表（前 10 所）
  - 基础录取概率估算
  - 院校基本信息查询

付费解锁（¥99~¥599）：
  - 完整志愿方案生成
  - "冲稳保"梯度表
  - 一键填报
  - 专家解读报告（实为模板生成）
  - 不限次数查询

附加变现：
  - 高考志愿规划师人工服务（¥1000~¥20000）
  - 一对一填报指导
```

## 三、典型产品技术特征对比

### 3.1 标准产品（70% 市场份额）

| 特征 | 实现方式 |
|------|----------|
| 核心技术 | SQL 数据库查询 + 硬编码规则 |
| AI 程度 | 无真实 AI，标签"AI"仅为营销用语 |
| 知识来源 | 公开分数线 + 招生计划数据 |
| 规则覆盖 | 5~10 条通用规则 |
| 章程理解 | 无，不读取招生章程原文 |
| 推理方式 | 固定流程，无多轮推理 |
| 输出形式 | JSON 一次性返回 |
| 实时性 | 年度更新，出分前灌库 |
| 个性化 | 仅按分数/选科/省份过滤 |
| 错误处理 | 无优雅降级，数据缺失即空结果 |

### 3.2 进阶产品（20% 市场份额，大厂出品）

| 特征 | 实现方式 |
|------|----------|
| 核心技术 | 数据库查询 + 轻量 ML 排序模型 |
| AI 程度 | 调用通用大模型做文案生成（非核心决策） |
| 知识来源 | 同上 + NLP 抽取招生章程关键句 |
| 规则覆盖 | 20~50 条规则，但多为关键词匹配 |
| 章程理解 | 基于正则提取的浅层理解 |
| 推理方式 | 单次 LLM 调用包装，无 Agent 框架 |
| 输出形式 | JSON + 模板化文案 |
| 实时性 | 可联网搜索补充，但准确性不可控 |
| 个性化 | 增加兴趣测评、职业倾向匹配 |
| 错误处理 | 基本容错，API 故障时回退静态方案 |

## 四、架构同质化的根源

### 4.1 为什么市面产品几乎一模一样

```
原因一：数据来源单一
  → 所有产品都依赖各省考试院公布的历年投档线
  → 数据相同 → 结果相近 → 功能趋同

原因二：技术门槛低
  → "输入分数→输出可报院校"是典型的 CRUD 开发
  → 任何有数据库经验的团队都能 1 周做出 MVP
  → 差异化只能靠营销和数据覆盖量

原因三：AI 标签滥用
  → 大部分产品的"AI"实质是 if-else 规则匹配
  → 少数产品接了通用大模型，但仅用于生成推荐文案
  → 核心决策逻辑仍是数据库查询

原因四：付费模式倒逼简化
  → 产品本质是卖数据报告，不是卖分析能力
  → 越复杂的分析 → 越难标准化定价 → 越难规模化
```

### 4.2 用户实际得到的产品

```
用户以为买了：
  「AI 深度分析 + 个性化推荐 + 精准录取预测」

用户实际得到：
  ┌─────────────────────────┐
  │  一个美化的 SQL 查询界面 │
  │  + 几条硬编码的通用规则  │
  │  + 一份模板生成的报告    │
  └─────────────────────────┘
```

## 五、与「高考志愿狙击手」的本质差异

| 维度 | 传统产品范式 | 高考志愿狙击手 |
|------|-------------|---------------|
| **核心定位** | 推荐志愿（"该报什么"） | 审查志愿（"有没有雷"） |
| **数据来源** | 公开分数线 + 招生计划 | 招生章程原文 + 分数线 + 招生计划 |
| **规则来源** | 人工编写的 5~10 条通用规则 | 200+ 高校章程的结构化提取 |
| **章程理解** | 无 | 双层检索：结构化规则 + 全文检索 |
| **推理方式** | 固定流程，单次查询 | Agent 多轮推理 + 工具调用 |
| **AI 参与度** | 标签化 AI（实际无 AI 参与决策） | 多模型调度，Agent 自主选择工具 |
| **输出形式** | JSON 一次性返回 | SSE 流式，展示推理过程 |
| **容错机制** | 数据缺失即无结果 | 多级降级：云端推理 → 规则引擎 → 通用兜底 |
| **联网能力** | 无或简单搜索 | 双层知识库 + 实时联网检索 |
| **个性化** | 分数/省份/选科 三维过滤 | 体检/单科/语种/选科/偏好 五维审查 |
| **透明度** | 黑盒概率 | 逐条展示推理过程 + 引用具体章程条款 |
