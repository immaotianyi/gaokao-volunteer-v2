# 高考志愿狙击手 — 工程实现机制说明

> 本文面向公众介绍系统整体技术架构和工作原理。
> 为保护用户数据安全和核心算法，部分实现细节已做抽象化处理。

---

## 一、系统架构总览

高考志愿狙击手采用 **前后端分离 + 容器化部署** 架构，由四个协同服务组成：

```
用户浏览器 (Vue 3 单页应用)
    │  HTTP/HTTPS + SSE 流式长连接
    ▼
前端网关层 (Nginx)
    │  静态文件托管 + API 反向代理
    ▼
后端服务层 (FastAPI)
    │  多 Worker 并发 + 异步 I/O
    ├──▶ 数据存储层 (持久化数据库 + 内存缓存)
    └──▶ AI 推理层 (云端大模型 API)
```

## 二、核心功能模块

### 2.1 志愿探雷器（Risk Scanner）

**功能定位**：对用户提交的志愿表逐条审查，发现因招生规则不符导致的退档风险。

**工作流程**：

```
用户输入 ──▶ 档案解析 ──▶ 知识检索 ──▶ AI 推理 ──▶ 风险报告
   │            │            │            │            │
   │       提取：分数      查询：      调用：       输出：
   │       体检状况        招生章程    大模型        DANGER
   │       单科成绩        规则库      逐条审查      WARNING
   │       语种选科                                   PASS
```

**审查维度（按优先级）**：

| 维度 | 风险等级 | 说明 |
|------|----------|------|
| 体检限制 | 🔴 DANGER | 色盲/色弱/视力不满足专业硬性要求 |
| 单科门槛 | 🟡 WARNING | 英语/数学/语文成绩未达专业最低线 |
| 语种限制 | 🔴 DANGER | 非英语语种考生报考仅限英语的专业 |
| 选科不符 | 🔴 DANGER | 选考科目不满足专业要求 |
| 低偏好风险 | 🟡 WARNING | 专业处于高调剂风险方向 |

---

### 2.2 捡漏雷达（Leakage Radar）

**功能定位**：发现因信息不对称导致的低竞争录取机会。

**捡漏策略矩阵（多维度交叉分析）**：

```
┌──────────────────────────────────────┐
│  策略维度        │  检测方法         │
├──────────────────┼───────────────────┤
│  新增专业        │  年际招生计划对比  │
│  大幅扩招        │  计划数增幅分析    │
│  新校区首招      │  院校校区字典匹配  │
│  中外合作        │  专业名称特征识别  │
│  选科稀缺        │  选科组合占比计算  │
│  纯净专业组      │  组内风险占比评估  │
└──────────────────┴───────────────────┘
```

**处理流程**：

1. **省份+科类过滤** — 精准定位考生所在省的招生计划
2. **联合主键构建** — 以"院校代码+专业组代码+专业代码"为唯一标识
3. **年际差分计算** — 对比上下两年招生计划，挖掘新增与扩招
4. **噪音过滤** — 剔除省外首次招生等非真实捡漏信号
5. **纯净组过滤** — 只保留无调剂风险的志愿组
6. **多维评分** — 综合多个维度给出 0-100 分的捡漏指数
7. **排序输出** — 按捡漏价值降序排列

---

### 2.3 AI 志愿顾问（Advisor）

**功能定位**：以对话形式为考生提供志愿填报咨询，支持流式输出和多轮对话。

**核心能力**：
- 基于招生章程知识库回答具体院校专业的规则问题
- 联网搜索补充本地未覆盖的高校信息
- 多轮上下文记忆，支持追问和澄清
- 流式输出（SSE），逐字推送给用户

## 三、知识工程体系

### 3.1 招生章程知识库

系统构建了一套**多层检索架构**，确保对招生章程规则的精准匹配：

```
检索请求 (大学+专业)
    │
    ├── 第一层：结构化规则精准匹配
    │   来源：经过人工标注和AI辅助提取的结构化规则库
    │   特点：O(1) 复杂度，精确到专业级别
    │   覆盖：核心高校的体检/单科/语种规则
    │
    ├── 第二层：章程全文模糊检索
    │   来源：教育部阳光高考平台招生章程原文
    │   特点：补充结构化未覆盖的条款
    │   覆盖：200+ 高校的完整章程文本
    │
    └── 第三层：联网实时搜索
        来源：互联网公开招生信息
        特点：动态补充本地未收录的高校
        标注：联网搜索结果标注"live"来源
```

### 3.2 数据层级说明

| 数据层 | 来源 | 覆盖范围 | 更新频率 |
|--------|------|----------|----------|
| 招生计划数据 | 各省教育考试院公开信息 | 多省多科类 | 年度更新 |
| 招生章程规则 | 教育部阳光高考平台 | 200+ 高校 | 年度校验 |
| 历史录取数据 | 历年公开录取统计 | 多年趋势 | 持续积累 |
| 体检指导意见 | 教育部官方文件 | 全国通用 | 政策同步 |

## 四、AI 推理架构

### 4.1 模型调度策略

系统采用 **多级推理调度**，根据任务复杂度和可用资源自动选择最优路径：

```
推理请求
    │
    ├── 云端大模型推理（首选）
    │   用于：复杂规则判断、联网章程解析
    │   特点：深度语义理解，准确率高
    │
    ├── 本地规则引擎（回退）
    │   用于：API不可用时的降级服务
    │   特点：基于结构化规则库的确定性推理
    │
    └── 通用规则兜底（最终保障）
        用于：所有路径均不可用时
        特点：教育部通用体检指导意见
```

### 4.2 Agent 推理框架

针对复杂审核场景，系统实现了一套**多轮推理 Agent 框架**：

```
第一轮：理解考生档案 + 志愿目标
    │
    ▼
第二轮：自主选择知识工具
    ├─ 招生章程检索工具
    ├─ 体检风险检查工具
    ├─ 单科成绩核验工具
    └─ 历史数据查询工具
    │
    ▼
第三轮：综合推理，逐条验证
    │
    ▼
输出：结构化审核报告
    ├─ 风险等级（DANGER/WARNING/PASS）
    ├─ 审核理由（引用具体条款）
    └─ 置信度评分
```

Agent 支持 **最大 N 轮** 工具调用，运行时可根据推理需要动态选择工具组合。

## 五、数据传输机制

### 5.1 SSE 流式推送

系统核心审查功能采用 **Server-Sent Events (SSE)** 协议实现流式数据推送。

**流式事件类型**：

| 事件类型 | 触发时机 | 数据内容 |
|----------|----------|----------|
| `thinking` | Agent 开始推理 | 当前推理步骤描述 |
| `tool_call` | Agent 查询知识库 | 工具名称 + 查询参数 |
| `tool_result` | 知识库返回结果 | 检索到的规则条款 |
| `conclusion` | 推理完成 | 结构化审核结果 |
| `done` | 全部完成 | 结束标记 |

**设计优势**：
- 用户可实时观察 AI 的推理过程，增强信任感
- 无需轮询，降低服务端压力
- 支持断线自动重连

### 5.2 数据安全

- 用户档案数据仅在当前会话中使用，不持久化存储个人信息
- 所有 API 通信支持 HTTPS 加密
- 云端推理仅发送脱敏的审查请求，不传输用户身份信息

## 六、前端技术方案

### 6.1 技术栈

- **框架**：Vue 3 + TypeScript
- **构建**：Vite
- **状态管理**：Pinia
- **路由**：Vue Router
- **样式方案**：CSS 原生变量 + 响应式布局

### 6.2 组件架构

```
App
 ├── ProfilePanel      — 考生档案面板（支持内联编辑弹窗）
 ├── RiskScanner       — 探雷器主界面（SSE 流式 + 统计条）
 ├── RadarBoard        — 捡漏雷达网格（筛选 + 排序 + 付费墙）
 ├── PaymentModal      — 支付弹窗（多模式兼容）
 └── Toast System      — 全局通知系统
```

### 6.3 容错设计

- **支付降级**：真实支付不可用时自动切换模拟模式
- **SSE 断线重连**：带指数退避的自动重连机制
- **API 容错**：请求失败时友好的错误提示和重试引导

## 七、部署方案

### 7.1 容器化部署

系统支持 Docker Compose 一键部署：

```
docker-compose.yml
 ├── frontend  — Nginx + Vue 3 静态文件
 ├── backend   — FastAPI (多 Worker)
 ├── database  — 持久化数据库
 └── cache     — 内存缓存
```

### 7.2 扩展能力

- 后端支持水平扩展（增加 Worker 数量）
- 缓存层支持独立部署
- 支持接入外部 CDN 加速静态资源
- 数据库支持主从复制和读写分离

## 八、能力边界说明

为保障用户权益，特此声明系统当前能力边界：

### 已覆盖

- 200+ 高校招生章程的体检/单科/语种规则
- 教育部通用体检指导意见完整兜底
- 多省多科类招生计划年际对比分析
- 增专业/扩招/新校区/中外合作等多维捡漏策略

### 当前局限

- 历史录取分数线数据依赖公开来源，部分院校年份数据不完整
- 非核心高校的章程规则可能未被结构化提取，需依赖联网搜索
- 估值模型基于统计规律推算，实际录取分数受多种因素影响
- 专业组内部调剂风险无法完全量化

### 使用建议

1. 系统审查结果作为参考，最终志愿填报决策请以各省教育考试院官方信息为准
2. 联网搜索结果标注"live"来源，建议用户二次核实
3. 新增专业的估值分数为统计推断，存在不确定性区间
4. 建议结合官方招生章程原文进行最终确认
